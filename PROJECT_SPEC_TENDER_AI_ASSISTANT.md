## 1. Краткое описание проекта

Необходимо разработать веб-сервис и Telegram-бота для автоматического поиска, фильтрации, анализа и отправки пользователям подходящих тендеров/закупок.

Проект предназначен для малого и среднего бизнеса, который хочет находить подходящие закупки, быстро понимать требования заказчика, оценивать целесообразность участия и получать краткие AI-выжимки по закупочной документации.

Система должна работать так:

1. Пользователь регистрируется через сайт или Telegram-бота.
2. Пользователь указывает свою нишу, ключевые слова, регионы, минимальную/максимальную сумму закупки, тип закупок и стоп-слова.
3. Система регулярно ищет подходящие закупки в открытых источниках.
4. Система фильтрует нерелевантные закупки.
5. AI-модуль делает краткий анализ закупки.
6. Пользователь получает подборку в Telegram и/или видит её в веб-кабинете.
7. Пользователь может сохранять закупки, ставить статусы, оставлять заметки, выгружать отчёты и формировать черновик коммерческого предложения/письма.

Проект должен быть реализован как MVP, но с архитектурой, которую можно масштабировать.

## 2. Главная цель продукта

Создать простой, понятный и коммерчески применимый сервис, который помогает малому бизнесу экономить время на поиске закупок и находить потенциально прибыльные тендеры.

Главная ценность продукта:

- не просто список закупок;
- а понятная ежедневная подборка;
- с кратким AI-анализом;
- с рекомендацией: “стоит участвовать / можно рассмотреть / сомнительно / не подходит”;
- с возможностью работать через Telegram и веб-кабинет.

## 13. Источники данных по закупкам

На MVP необходимо реализовать архитектуру источников данных так, чтобы можно было подключать разные источники.

### 13.1. Интерфейс источника

Каждый источник должен реализовывать единый интерфейс:

```ts
interface TenderSource {
  name: string;

  search(params: TenderSearchParams): Promise<RawTender[]>;

  getDetails?(externalId: string): Promise<RawTenderDetails>;

  getDocuments?(externalId: string): Promise<RawTenderDocument[]>;
}
```

### 13.2. Первые источники для MVP

На MVP нужно реализовать минимум один источник.

Возможные источники:

- открытые данные ЕИС;
- публичные страницы поиска закупок;
- выгрузки/фиды, если доступны;
- ручной импорт CSV/XLSX на первом этапе.

Если автоматический источник сложен, MVP допускает ручной импорт закупок через админ-панель, но архитектура должна быть готова к автоматическому сбору.

### 13.3. Юридическое ограничение

Система должна использовать только разрешённые и публично доступные источники данных.

Запрещено:

- обходить авторизацию;
- обходить капчи;
- обходить платный доступ;
- обходить технические ограничения площадок;
- нарушать правила использования сайтов;
- использовать закрытые API без разрешения.

## 16. Алгоритм поиска и фильтрации

### 16.1. Общий процесс

1. Worker берёт активные профили поиска.
2. Для каждого профиля вызывает источники данных.
3. Источники возвращают сырые закупки.
4. Система нормализует данные.
5. Система проверяет, была ли закупка уже сохранена.
6. Система сохраняет новую закупку.
7. Система считает релевантность закупки для профиля.
8. Если закупка подходит, создаётся связь `TenderMatch`.
9. Если включён AI-анализ, закупка отправляется в очередь AI-анализа.
10. После анализа пользователь получает уведомление.

### 16.2. Правила фильтрации

Закупка подходит, если:

- содержит хотя бы одно ключевое слово;
- не содержит стоп-слова;
- соответствует региону;
- соответствует диапазону цены;
- не истёк срок подачи заявки;
- соответствует выбранному типу закупки;
- не была ранее скрыта пользователем.

### 16.3. Релевантность

Для каждой закупки нужно рассчитывать score от 0 до 100.

Пример расчёта:

- совпадение в названии: +30;
- совпадение в описании: +15;
- совпадение по региону: +15;
- цена в заданном диапазоне: +15;
- срок подачи больше 3 дней: +10;
- нет стоп-слов: +10;
- совпадение по заказчику: +5.

Если score ниже 40, закупку не показывать по умолчанию.

## 17. AI-анализ закупки

AI-анализ должен превращать сложное описание закупки в понятную карточку.

### 17.1. Что должен делать AI

Для каждой подходящей закупки AI должен сформировать:

1. Краткое описание.
2. Что нужно поставить или выполнить.
3. Кто заказчик.
4. Сумма закупки.
5. Срок подачи.
6. Основные требования.
7. Возможные риски.
8. Какие документы могут понадобиться.
9. Насколько закупка подходит профилю пользователя.
10. Рекомендация:
    - стоит участвовать;
    - можно рассмотреть;
    - сомнительно;
    - не подходит.

### 17.2. Формат AI-ответа

AI должен возвращать строго JSON.

Пример:

```json
{
  "summary": "Закупка на поставку лицензий Microsoft Office для муниципального учреждения.",
  "whatToDeliver": "Поставка программного обеспечения и передача лицензий.",
  "customer": "Муниципальное учреждение ...",
  "requirements": [
    "Поставка в течение 10 рабочих дней",
    "Наличие документов, подтверждающих право поставки",
    "Соответствие техническому заданию"
  ],
  "risks": [
    "Короткий срок поставки",
    "Возможны требования к официальному происхождению лицензий"
  ],
  "documentsNeeded": [
    "Коммерческое предложение",
    "Декларация соответствия",
    "Учредительные документы",
    "Подтверждение полномочий подписанта"
  ],
  "fitScore": 82,
  "recommendation": "worth_participating",
  "recommendationText": "Закупка выглядит подходящей, если компания может подтвердить легальность поставляемых лицензий."
}
```

### 17.3. Требования к AI

- AI не должен выдумывать факты.
- Если данных недостаточно, AI должен писать “не указано”.
- AI не должен давать юридические гарантии.
- AI не должен утверждать, что закупка точно прибыльная.
- AI должен использовать осторожные формулировки.
- AI-ответ должен валидироваться через Zod-схему.
- Если AI вернул невалидный JSON, нужно повторить запрос один раз.
- Если повтор не помог, сохранить ошибку и показать базовый анализ без AI.

## 33. Безопасность

### 33.1. Авторизация

- JWT access token.
- Refresh token желательно, но для MVP можно session cookie.
- Пароли хранить только в виде hash.
- Использовать bcrypt или argon2.
- Админские API закрыть ролью `ADMIN`.

### 33.2. Telegram

- Telegram ID нельзя считать полноценной авторизацией для веба без привязки.
- Для связи Telegram и веб-аккаунта использовать одноразовый токен.
- Токен должен иметь срок жизни 10–15 минут.

### 33.3. API

- Валидация всех входящих данных через Zod.
- Rate limit на публичные endpoint.
- Защита от SQL injection через Prisma.
- Логирование ошибок без вывода секретов.
- Не хранить API-ключи в коде.

### 33.4. ENV

Все секреты должны быть в `.env`.

Пример:

```env
DATABASE_URL=
REDIS_URL=
JWT_SECRET=
TELEGRAM_BOT_TOKEN=
APP_URL=
API_URL=
AI_PROVIDER=
OPENAI_API_KEY=
OLLAMA_BASE_URL=
ADMIN_EMAIL=
```

## 41. Критерии готовности MVP

MVP считается готовым, если:

1. Пользователь может зарегистрироваться.

8. Пользователь получает закупки в Telegram.
9. Пользователь может добавить закупку в избранное.
10. Пользователь может поменять статус закупки.
11. Администратор видит пользователей и закупки.
12. Проект запускается через Docker Compose.
13. Есть README.
14. Есть `.env.example`.
15. Нет критических ошибок при базовом сценарии.
